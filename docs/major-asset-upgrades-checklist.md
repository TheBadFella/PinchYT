## Notes

- Tailwind CSS v4 removes the old `@import "tailwindcss/base|components|utilities"` entrypoint style in favor of `@import "tailwindcss"`.
- Tailwind CSS v4 no longer auto-detects `tailwind.config.js`; this repo now uses CSS-native configuration instead.
- Tailwind CSS v4 has browser baseline changes and some utility behavior changes. A manual UI pass is required even if the build succeeds.
- The Dockerfiles currently manage the Node runtime for asset compilation independently of `mix.exs`, so the major asset upgrade needs a matching Docker update.
- `docker compose run --rm phx mix test` passes after the upgrade.
- `docker compose run --rm phx mix check` passes after the upgrade.

## Tailwind v4 Status

The repo is now using native Tailwind v4 CSS configuration.

- `assets/css/app.css` defines the custom Tailwind tokens with `@theme`
- Tailwind source detection is declared in CSS with `@source`
- `assets/tailwind.config.js` has been removed
- `lib/pinchflat_web/components/core_components.ex` now renders icons inline from vendored SVGs instead of relying on Tailwind plugins

## Findings From Audit

- No application code currently uses the custom screen prefixes `2xsm:`, `xsm:`, or `3xl:` outside `assets/tailwind.config.js`, so they were removed
- No application code currently uses the custom `phx-*:` Tailwind variants, so they were removed
- The `@tailwindcss/forms` plugin was removed from both CSS and package dependencies
- Heroicons and Simple Icons are no longer generated by Tailwind plugins; they are rendered inline by `PinchflatWeb.CoreComponents.icon/1`
- Most of the previous JS config was dead baggage; the live app only needed colors, radii, shadows, title text sizes, z-index aliases, and font setup migrated into `@theme`
